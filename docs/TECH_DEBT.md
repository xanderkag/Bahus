## 🔴 Критические ограничения прототипа (намеренно не исправляются)

### 1. Ephemeral Filesystem (файлы теряются при рестарте Railway)
- **Симптом:** Загруженные файлы хранятся на диске Railway-контейнера. При каждом рестарте или деплое они пропадают.
- **Последствие:** Старые импорты (`queued`) после рестарта не могут быть повторно отправлены в n8n — файл физически удалён.
- **Для прототипа:** Допустимо. Файл загружается → немедленно уходит в n8n → в БД хранятся результаты. До перезапуска всё работает.
- **Решение для продакшн:** Supabase Storage или Railway Volumes (persistent disk).

### 2. Нет аутентификации
- API открыт для любых запросов без токена.
- **Для прототипа:** Допустимо — демо-среда, не публичный URL.
- **Решение для продакшн:** JWT/API-key middleware в `postgres_api.py`.

### 3. Один поток обработки HTTP
- Бэкенд (`ThreadingHTTPServer`) не масштабируется под нагрузку.
- **Для прототипа:** Достаточно для 1-5 одновременных пользователей.

## 🔵 Технический долг (для продакшн-версии)

| Задача | Приоритет | Описание |
|---|---|---|
| Persistent file storage | Высокий | Supabase Storage вместо локального диска |
| Аутентификация | Высокий | JWT middleware, управление пользователями |
| HTTPS/домен | Средний | Кастомный домен для Vercel (обход РКН) |
| Retry-логика n8n | Средний | Повтор отправки при таймауте n8n |
| Unit-тесты API | Низкий | Pytest для критических маршрутов |
| Webhook-подпись | Низкий | HMAC-верификация входящих вебхуков от n8n |
