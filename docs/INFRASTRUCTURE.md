# Инфраструктура Bakhus

Документ описывает устройство боевого сервера проекта Bakhus, развернутого в Yandex Cloud, а также механизмы CI/CD и безопасность интеграций.

## 1. Схема железа (Yandex Compute Cloud)
Проект развернут на единой недорогой (Preemptible) виртуальной машине.

- **IP-адрес**: `111.88.144.93` (Публичный)
- **Пользователь (SSH)**: `bakhus`
- **Рабочая директория**: `/home/bakhus/bakhus/`
- **Авторизация**: Доступ жестко ограничен по SSH-ключу `id_ed25519` (парольный вход отключен).

## 2. Docker Compose Стек
Продакшн поднимается с помощью файла `docker-compose.prod.yml`. Он оркестрирует три основных контейнера:

### A. Nginx (`bakhus-frontend`)
Служит единой точкой входа.
- Слушает `port 80`.
- Раздает статические файлы из `index.html` и `/src`.
- Проксирует все запросы, начинающиеся с `/api/`, во внутренний контейнер `postgres-api` на порт `8078`.
- Изменения: Увеличен `client_max_body_size` до `50M` для поддержки загрузки тяжелых PDF прайсов.

### B. Python API (`bakhus-postgres-api`)
Вся бизнес-логика.
- Работает из `scripts/postgres_api.py`.
- **Фоновые потоки**: Встроен `cleanup_worker`, который каждые 60 минут сканирует `UPLOADS_DIR` и удаляет файлы старше 7 дней, предотвращая переполнение диска.
- Отвечает за коммуникацию с `n8n` по протоколу HTTP(s).

### C. База Данных (`bakhus-postgres`)
Надежное хранилище `PostgreSQL 15-alpine`.
- Данные монтируются в Persistent Volume: `bakhus_bakhus-postgres-data`. При перезапуске ВМ данные никуда не исчезают.
- API контейнер защищен retry-логикой, так что если БД запускается дольше обычного, API будет ожидать соединения.

## 3. Автоматизация деплоя (CI/CD)
Процесс выкладки полностью автоматизирован и базируется на **GitHub Actions**.

1. Любой коммит в ветку `main` запускает процесс `.github/workflows/deploy.yml`.
2. Платформа GitHub авторизуется на сервере 111.88.144.93, используя сохраненные секреты.
3. Актуальный код копируется на сервер. Загруженные пользовательские файлы игнорируются при копировании, чтобы не было потерь данных.
4. Выполняется `docker-compose -f docker-compose.prod.yml up -d --build`, который незаметно обновляет только те контейнеры, чьё содержимое изменилось.

**Задействованные GitHub Secrets:**
- `SERVER_IP`
- `SERVER_USERNAME`
- `SSH_PRIVATE_KEY`

## 4. Специфика интеграции n8n (CORS Proxy)
Ранее архитектура подразумевала вызов n8n-вебхука напрямую из браузера, что вызывало ошибки CORS (браузеры режут кросс-доменные запросы из соображений безопасности).

**Текущее элегантное решение:**
Браузер отправляет запрос локально на свой же домен (`/api/proxy/n8n`).
Контейнер `postgres-api` перехватывает этот запрос, парсит бинарные данные, извлекает целевой URL `target_webhook_url`, и "под капотом" шлет запрос напрямую в n8n через серверный `requests.post`. 
Это гарантирует обход CORS, так как n8n даже не догадывается, что первоисточник запроса — браузер, и не требует никакой сложной конфигурации вебхук-узлов в n8n на стороне клиента.
